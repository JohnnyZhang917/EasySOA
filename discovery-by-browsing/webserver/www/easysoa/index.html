Introduction

Welcome to EasySOA Core & Light!

First time user? Take the tour!

Note: Before using this demo, please set your browser's proxy to localhost:8081 and change the proxy exceptions (unless you're using our virtual machine): please remove any localhost or 127.0.0.1, and add instead *github.com and *akamai.net. If you're asked for a user/password, just type Administrator for both. In case something goes wrong during the demo, you can check the troubleshooting page.

Step 1
Discover services in your intranet

More on discovery by browsingFor tests purposes, this demo packs a mockup intranet of a company named Pure Air Flowers. Let's browse it using the Service Discovery. From there, as you visit pages, you may see a service appearing in the left menu, meaning that EasySOA has just found it. You're then two clicks away to register this service: just select "PureAirFlowers Orders", then click "Save Service", and you're done!

Step 2
Explore all services

More on the service registryNow, let's have a look to our registered service, that you will find in the Service Registry. Once you're logged in, navigate to Workspaces > Services. You will find here the service you've just registered. If you click on it, all metadata are now here, business and technical. You may override most of these metadata, and fill new ones if you want to document your service.

There are various ways to browse services: other than the default navigation tree, there is of course a search form, but also what is called "virtual navigations". These can be accessed from the 3 last tabs on the left menu: you can use them to browse through services in various fashions, each one being more appropriate for a specific user profile (business user, developer, IT admin). In the long term, we plan on making the interface fully adapted to each type of user, hiding for exemple the technical metadata from business users.

Step 3
Call a service

Now, thanks to EasySOA, the service you've just found can be easily called. While you're on the service registry, notice the "Call it!" button on your service page. Click on it! The interface you now see has been generated from the registered definition of the service. Here you can enter the input parameters, and start a request. Is the service answering "10"? It's working!

This call is actually made in an SOA sandbox that proxies, monitors and protects actual services. In this case, it prevents the user from using them excessively. You can test it by calling the service repeatedly : an error comes once you've called the service more than 10 times in 2 minutes. The calls are then blocked for about 2 minutes! And if you call a service more than 10 times a second, EasySOA electrocutes you - just kidding.

Conclusion

For now, EasySOA is a demo provided with all imaginable disclaimers... But later, lots of features will be added, from collaborative features and true proxy generation to an all-in-one architecture. For more advanced features, including services monitoring and architecture files import, please see the second demo scenario.

This scenario is based on another demo SOA, called "Smart Travel". This time, we will take the role of a developer who wants to map the SOA contents.

Step 1
Import the SOA architecture

More on SCA importFirst, we are going to import the whole SOA architecture into the service registry. In order to do that, a feature called "SCA import" is able to analyze the definition of the SOA components. From the service registry, create a new application by going to the "Service registry" document and choosing "New Application Implementation" ; create it with "Smart Travel" as the title and "http://localhost" for the URL. Then, import the SCA file through the "SCA data import" page, available from the top menu. Choose the file located in "exemples/smart-travel-mock-services.composite" (in your EasySOA folder), then select the newly created application for the import, and go!

Step 2
Explore the services

You can now browse through various services in the left menu. Only two of them are in the "Smart Travel" application, since others are external services, which are only referenced from your application. All the imported data can be useful to help developers and architects, even more once documentation files are attached to the services (you can do it by editing any document).

Now, what if a member of the IT staff wants to know what really happens on the network?

Step 3
SOA Monitoring

More on discovery by monitoringIn EasySOA, HTTP monitoring can be used both to discover new services, and simply to monitor the information system. You will eventually have the possibility to set up monitoring probes anywhere. For now, EasySOA embeds a simple HTTP monitoring proxy. You can test this feature by using EasySOA Light, since monitoring is enabled on it by default. From the Service Registry, find the "trip" service (somewhere in the "Smart Travel" app), and click the "Call It!" button: all requests you will send from there will be intercepted by the embedded proxy, that will then send information about this service to the Service Registry. (Note: make sure you type a number in the third argument)

Step 4
Explore the service Pt. 2: data correlation

Notice that thanks to monitoring and SCA import, we get two different ways to find the same services, allowing to gathering complementary information. If you add to them the discovery by browsing tool (see demo scenario #1), they all turn into an interesting way to gather crss-cutting data regarding the information system. In the service registry, the correlation between these discovery methods is represented by three icons shown on every document page: . For exemple, browse to the "Create Summary" service to see how these icons are enabled for the browsing and monitoring discovery. There, you will also see that the "call count" property has been increased!

Step 5
The dashboard

More on the dashboardSince we are storing in EasySOA how many times each service has been called, why not have a way to provide all these information on a synthetic page? The dashboard, available from the top menu, allows you to organize various widgets on the same page. One of those widgets is called "EasySOA Stats". Add it to see a list of services showing how many called has been made on each one.

Conclusion

These two demos show you various ways to use and interact with EasySOA and its cross-cutting model. Some facets of the solutions have not been shown yet though: since users of various profiles will be able to use the solution, why not make them work together, thanks to collaborative features? What if the business user could directly send requests to the developers by "commenting" a service? The ambitions of EasySOA go further than what you just saw, so stay tuned. Thanks for testing EasySOA!
