Introduction

Welcome to EasySOA Core & Light!

First time user? Take the tour!

Note: Before using this demo, please set your browser's HTTP proxy to localhost:8081 (enable only the HTTP protocol) and remove any proxy exceptions. In case something goes wrong during the demo, you can check the troubleshooting page.

Step 1
Discover services in your intranet

In order to login as a Business User, use Sophie as username/password.

More on discovery by browsingFor tests purposes, this demo packs a mockup intranet of a company named Pure Air Flowers. Let's browse it using the Service Discovery. From there, as you visit pages, you will see services appearing in the left menu, meaning that EasySOA has just found them. You're then two clicks away to register this service: just select "Orders service", then click "Save Service", and you're done!

Step 2
Explore all services

More on the service registryNow, let's have a look to our registered services. You will find them in the Service Registry. Once you're logged in, navigate to Workspaces > Service Registry. You will find here a "Default Application" document what you've just registered. If you open the "Orders service" service, all metadata are now here, business and technical (since you're logged as a business user, you'll have to unfold the "Technical information" block to see its data, but everything is here). You can edit everything, and add more information if you want to document your service.

Now what about some communication with other users? More than allowing service documentation to be collaborative, the registry provides real social features. Explore the tabs above your service. You can:
  - Add comments, to talk with other users about the service (try to post a bug report to the developers!)
  - Start a workflow
  - View the document changes history, etc.

Step 3
Call a service

Thanks to EasySOA, the service you've just found can be easily called. While you're on the service registry, notice the "Call it!" button on your service page. Click it! The interface you now see has been generated from the registered definition (WSDL) of the service. Here you can enter the input parameters, and start a request.

The services calls are actually made in an SOA sandbox that proxies, monitors and protects actual services. In this case, it prevents the user from using them excessively. You can test it by calling the service repeatedly : an error comes once you've called the service more than 10 times in 2 minutes. The calls are then blocked for 2 more minutes! ...And if you call a service more than 10 times a second, EasySOA electrocutes you (just kidding).

Conclusion

This should have gotten you a glimpse of what we mean by making EasySOA a "light", "open" and "collaborative" solution. All features you just tried here work out of the box, since all data used to fill the registry are discovered through the use of a simple browser.

Right know, EasySOA is a demo provided with all imaginable disclaimers... But later, lots of features will be added, from the creation of custom services to an all-in-one architecture.

For more advanced features, including services monitoring and architecture files import, please see the second demo scenario.

This scenario is based on another demo SOA, called "Smart Travel". This time, we will take the role of a developer who wants to map the SOA contents.

Step 1
Import the SOA architecture

In order to login as a Software architect, type Ted as username/password.

More on SCA importFirst, we are going to import the whole SOA architecture into the service registry. In order to do that, a feature called "SCA import" is able to analyze the definition of the SOA components. From the service registry, create a new application by going to the "Service registry" document and choosing "New Application Implementation" ; create it with "Smart Travel" as the title and "http://localhost" for the services URL. Then, import the SCA file through the "EasySOA > SCA data import" page, from the top menu. Choose the file located in "examples/smart-travel-mock-services.composite" (in your EasySOA folder), then select the newly created application for the import, and go!

Step 2
Explore the services

You can now browse through various services in the left menu (you might need to click on its refresh icon). All the imported data can be useful to help developers and architects, even more once documentation files are attached to the services (you can do it by editing any document). All of the collaborative features are again available.

Now, what if a member of the IT staff wants to know what really happens on the network?

Step 3
SOA Monitoring

In order to login as a member of the IT Staff, type Arnold as username/password.

More on discovery by monitoringIn EasySOA, HTTP monitoring can be used both to discover new services, and simply to monitor the information system. You will eventually have the possibility to set up monitoring probes anywhere. For now, EasySOA embeds a simple HTTP monitoring proxy. You can test this feature either by using a "Call it!" button from the registry, or by using the direct link to the Service scaffolder, on the left of this page. All requests you will send will be intercepted by the embedded proxy, that will then send information about this service to the Service Registry. (Note: make sure to type a number when a field asks for it)

Step 4
Explore the service Pt. 2: data correlation

Notice that thanks to monitoring and SCA import, we get two different ways to find the same services, allowing to gathering complementary information. If you add to them the discovery by browsing tool (see demo scenario #1), they all turn into an interesting way to gather crss-cutting data regarding the information system. In the service registry, the correlation between these discovery methods is represented by three icons shown on every document page: . For exemple, browse to the "Create Summary" service to see how these icons are enabled for the browsing and monitoring discovery. There, you will also see that the technical "call count" property has been increased!

Step 5
The dashboard

More on the dashboardSince we are storing in EasySOA how many times each service has been called, why not have a way to provide all these information on a synthetic page? The dashboard, available from the top menu, allows you to organize various widgets on the same page. One of those widgets (in the Utilities category) is called "Service Stats". Add it to see a list of services showing how many called has been made on each one.

Conclusion

These two demos show you various ways to use and interact with EasySOA and its cross-cutting model. But the ambitions of EasySOA go further than what you just saw: services creation, more collaborative features, but also the configuration of the monitoring component, an all-in-one architecture... So stay tuned! Thanks for testing EasySOA!
